Part 1: Ugly math, ugly code

As I embarked on my journey to learn the math side of machine learning, all of the blog posts seemed to point to linear algebra as the starting point. The problem was, nothing I read made it immediately clear how linear algebra played a role in machine learning.

Worse yet, the whole discussion of vectors, matrices, linear combinations, and linear transformations seemed completely disconnected from my layman‚Äôs understanding of machine learning.

The Khan Academy videos on linear algebra are quite tedious and I didn‚Äôt feel I was getting anywhere. 3blue1brown‚Äôs LA series is much more engaging and lucid, but I was still getting bogged down in the theory without seeing the application.

Needless to say, it was pretty slow going.

My linear algebra a-ha moment

It wasn‚Äôt until I switched gears and decided, on a whim, to tackle linear regression that linear algebra really started to click for me. On a practical level, code that uses linear-algebraic methods simplifies work for the computer by orders of magnitude, making it possible to process massive datasets‚Äîand process them rapidly. This is obviously a critical requirement in the age of Big Data.

And on a conceptual level, it gave me my first mathematical a-ha moment:

By manipulating matrices and vectors, you can achieve the same outcome as a loop without explicitly looping‚Äîa loopless loop.

That is why linear algebra is the cornerstone of machine learning.

The only thing is, no single resource I found on the internet seemed to really clarify the mathematical and programmatic aspects at the same time without getting too abstract on the math side of things.

As a self-taught and formerly math-phobic coder, I needed a guide that progressed from from inelegant code (which I could understand) and inelegant math to (mind-blowingly) elegant math, which would then lay the groundwork for writing extremely elegant‚Äîand performant‚Äîcode (which is incomprehensible without understanding the math).

This is that guide, created from my notes from Week 1 of my machine learning journey. I‚Äôve split it up into multiple posts, since it‚Äôs quite long.

How to make a computer explode

I always thought that for-loops were simply a fact of life.

While I understood the basic principle behind stochastic gradient descent, I had never implemented it myself before. If I had tried my hand at it before learning the math involved, I probably would have come up with this monster:

# DON'T TRY THIS

samples = [ [ ( list of lists) ] ]
actual = [ ... ]
def gradient_descent(samples, actual, epochs, learning_rate):
    weights = [0 for feature in samples[0]] # 0 to start
    for _ in range(epochs):
        for j, weight in enumerate(weights):
            summation_part_of_cost = 0
            for i, sample in enumerate(samples):
                sum_of_sq_errs = (sum(hypothesis(sample, weights)) - actual[i]) ** 2
                summation_part_of_cost += sum_of_sq_errs * sample[j]
            partial_deriv = (1 / (2 * len(samples))) * summation_part_of_cost
            weights[j] -= learning_rate * partial_deriv
    return weights

# I REALLY HOPE YOU DIDN'T TRY THIS

Okay, maybe not. (It would be pretty hard to write this without understanding the math involved.)

Count those loops‚Äîthree, to be exact! Terrifying. Now, being handy with Python, I could probably have calculated partial_deriv in one line with an even more terrifying list comprehension, just to show off:

partial_deriv = (1 / (2 * len(samples))) * sum([sample[j] * (sum(hypothesis(sample, weights)) - actual[i]) ** 2) for i, sample in enumerate(samples)])

...which shaves off four lines, at the expense of all readability. But bigger problems remain:

  • The time complexity of this program is off the charts. In Big-O time complexity terms, it is at least \(O(n^3)\), if not more, which is cubic time, or (literally) exponentially horrible.
  • It doesn‚Äôt even work. Even with a dataset of unremarkable size, you‚Äôre bound to get a RuntimeWarning: overflow encountered in square error that can‚Äôt be avoided even if you set the learning_rate to an impractically small value like 0.00001. Trust me, I‚Äôve tried.

Is there any way beyond this impasse?

Linear regression: A basic overview

Basically, linear regression is used to make predictions about a thing based on its characteristics, assuming that

  • that there is some correlation among those characteristics and
  • that you have plenty of data about other things.

The intuition here can be explained with middle school algebra. Imagine you know the square footage and price of 200 houses and you want to estimate the price of a house with a given square footage.

Obviously, this is an oversimplified correlation for the sake of example. (And for some reason, everyone seems to explain this concept using houses, so why reinvent the wheel?)

If you were to make a scatter plot of that data, with the area along the x-axis and the price along the y-axis, the pattern might roughly look like it follows a line‚Äînot perfectly linear, but linear enough to predict the price of a house with \(x\) square footage. You can use linear regression to work backward from the data to determine this line, the line of best fit.

In middle school algebra, lines are written in the form

$$ y = \color{magenta}{m}\color{teal}{x} + \color{orange}{b} $$

where \(\color{teal}{x}\) is the input, \(\color{magenta}{m}\) is the slope of the line, \(\color{orange}{b}\) moves the line up and down on the graph, and \(y\) is the height of the line at point \(\color{teal}{x}\).

Our house problem can also be framed as a line, where \(\color{teal}{x}\) is the square footage, which influences the price by some value \(\color{magenta}{m}\), to which we add some kind of base price to bring us to the final price, \(y\).

Well, that was easy enough, right?

Multiple linear regression: Because more is better (or something)

In the real world, of course, area is not the only factor that decides the price of a house. There are many others. Can we still adapt our middle school equation to this problem if each house has 3 features‚Äîsay, area, nearby property values, and age of the building?

$$ y = \color{magenta}{m}\color{teal}{x} + \color{magenta}{n}\color{teal}{z} + \color{magenta}{o}\color{teal}{a} + \color{orange}{b} $$

We can, but it‚Äôs messy. (It‚Äôs also no longer a line, but let‚Äôs ignore that for now.) First, let‚Äôs rewrite that ‚Äúbase price‚Äù as \(b \cdot 1\).

$$ y = \color{magenta}{m}\color{teal}{x} + \color{magenta}{n}\color{teal}{z} + \color{magenta}{o}\color{teal}{a} + \color{magenta}{b}\cdot\color{teal}{1} $$

This gives us a nice symmetry: Notice that all of the teal variables are features, which are multiplied by their degree of influence (called a coefficient in statistics, or a weight in machine learning). When you add all these together, you get the price of the house.

This is called multiple linear regression. Most people wouldn‚Äôt skip directly to multiple LR after introducing single LR, but single LR is pretty easy to digest if you can understand high school calculus (derivatives), so it didn‚Äôt level up my math knowledge.

Now, let‚Äôs code our equation, putting all the feature values into a list features and the weights into another list weights. In Python, in increasing order of elegance, we can write the following:

# Using completely random numbers just to show the code
features = [1, 3, 4, 5] # 3 features; first value is the ‚Äúdummy feature‚Äù
weights = [0.6, 0.2, 0.1, 0.75]

# Brute-force addition
price = features[0] * weights [0] + features[1] * weights [1] + features[2] * weights [2] + features[3] * weights [3]

If we use \(x\)s to denote our features, \(Œ∏\)s to denote our weights, and subscript numbers to denote the position of each item in a list (series), then we can rewrite our equation in a slightly more organized way:

$$ h_Œ∏(x) = \color{magenta}{Œ∏_0}\color{lightgray}{x_0} + \color{magenta}{Œ∏_1}\color{teal}{x_1} + \cdots + \color{magenta}{Œ∏_n}\color{teal}{x_n} $$

which just happens to be the generalized form of linear regression‚Äîgeneral in the sense that it can accommodate any number of features, whether that‚Äôs 1 or 1,000.

Here, \(x\) is the collection of all feature values \(\color{teal}{x_1}\) through \(\color{teal}{x_n}\), where \(n\) is the number of features. (And \(x_{1000}\) actually isn‚Äôt too crazy in terms of real-world datasets!) It also includes the dummy feature \(\color{lightgray}{x_0} = 1\). Likewise, \(Œ∏\) is the collection of all weights, including \(\color{magenta}{Œ∏_0}\), the ‚Äúbase price‚Äù in our example. In machine learning, this is called the bias value.

Finally, the function notation \(h_Œ∏(x)\) indicates that this is the hypothesis for item (house) \(x\) given the collection of weights \(Œ∏\).

How to Python in math (Lesson 1)

If you know anything about programming, you know that the last line of code above is no way to write a program. Accommodating 100 features would be a chore, and accommodating a variable number of features would be impossible. Naturally, we would use the magic of iteration:

# Super-basic iteration over the lists
price = 0
for j in len(features):
  price += features[j] * weights[j]

No one actually writes the linear regression formula like this, but if you wanted to, you could express the above code in math using a summation:

$$ h_Œ∏(x) = \sum_{j=0}^n \color{magenta}{Œ∏_j}\color{teal}{x_j} $$

That Greek letter \(\sum\) (sigma) means summation. Basically, run a for-loop that adds the result of the following expression for each sample (house) starting at \(j=0\) until \(j=n\).

If you know Python better than you know math (as I did), then you might try further refactoring the code:

# Functional programming version
price = 0
for f, w in zip(features, weights):
  price += f * w

# More Pythonic version of the above
price = [f * w for f, w in zip(features, weights)]

I didn‚Äôt realize this when I wrote the first draft of this post, but even expressing the simple Python function zip(x, y) in math requires linear algebra. I‚Äôll get back to the non-fancy version of linear regression in just a minute, but for sake of thoroughness, if \(x\) and \(Œ∏\) are both vectors, then

$$ \begin{align} \vec x = \begin{bmatrix} x_0 \\ x_1 \\ \vdots \\ x_n \end{bmatrix} & \qquad{} \vec Œ∏ = \begin{bmatrix} Œ∏_0 \\ Œ∏_1 \\ \vdots \\ Œ∏_n \end{bmatrix} & \quad{} z(\vec x, \vec Œ∏) = \vec Œ∏\vec x \end{align} $$

Similarly, we can rewrite our equation as a function in Python, too. Let‚Äôs leave vectors aside for now, but keep the zip and encapsulate it into a function, because it‚Äôs clean and easy to understand.

def hypothesis(features, weights):
    return [f * w for f, w in zip(features, weights)]

Einstein notation: Making math less confusing by making it more confusing

This is all great if there‚Äôs only one \(x\) (house). But we will need tons of houses to make a decent prediction. Our list houses needs to be changed into a list of lists. For the sake of example, if we had three houses and three features, houses would look like this (remember the dummy feature):

houses = [[1, 7, 4, 3]
          [1, 6, 7, 3],
          [1, 2, 9, 6]] # random values

This also allows us to refer to specific features of specific houses using two indexes, houses[i][j]. How do we do this in math, though? Enter Einstein notation.

$$ h_Œ∏(x^i) = Œ∏_0x_0^i + Œ∏_1x_1^i + \cdots + Œ∏_nx_n^i $$

The superscript numbers here aren‚Äôt exponents. You would think Einstein, of all people, could come up with something less confusing, but that is the convention, so it‚Äôs important to become familiar with it.

Just remember that in linear regression, there are conventional choices for the variable names. \(i\) denotes the \(i\)th house, and \(j\) the \(j\)th feature.

$$ x^{\color{orange}{i \textrm{th sample}}}_{\color{blue}{j \textrm{th feature}}} $$

So if we were to start describing each hypothesis for our dataset individually,

$$ h_Œ∏(x^i) = \left\{\begin{array}{ll} h_Œ∏(x^0) = Œ∏_0x_0^0 + Œ∏_1x_1^0 + \cdots + Œ∏_nx_n^0 \\ h_Œ∏(x^1) = Œ∏_0x_0^1 + Œ∏_1x_1^1 + \cdots + Œ∏_nx_n^1 \\ h_Œ∏(x^2) = Œ∏_0x_0^2 + Œ∏_1x_1^2 + \cdots + Œ∏_nx_n^2 \\ h_Œ∏(x^3) = Œ∏_0x_0^2 + Œ∏_1x_1^2 + \cdots + Œ∏_nx_n^2 \end{array}\right. $$
hypothesis(houses[0], weights)
hypothesis(houses[1], weights)
hypothesis(houses[2], weights)
hypothesis(houses[3], weights)

hyps = [hypothesis(houses[house], weights) for house in houses]

Cost function: How accurate are our predictions?

Seeing as the goal of linear regression is to come up with a line that best fits the data, we need some way to evaluate a line‚Äôs goodness of fit to the data. One measure of that is the mean squared error.

Error

Error is how far the prediction for one sample (house) is from its actual value.

$$ \begin{align} \textrm{error}_i &= \textrm{prediction}_i - \textrm{actual}_i\\ &= \hat{Y}_i - Y_i \end{align} $$

Note: The subscript \(i\) here is not Einstein notation, because these are just lists of values, not a ‚Äúspreadsheet‚Äù of rows and columns. The Einstein notation in this discussion of linear regression only applies to \(x\).

\(\hat Y\) is read ‚ÄúY-hat,‚Äù which is just a statistical convention. It can be substituted with our function \(h_Œ∏(x^i)\).

$$ \textrm{error}_i = h_Œ∏(x^i) - Y_i $$
actual_values = [3, 4, 5] # 3 houses; random values
weights = [0.6, 0.2, 0.1, 0.75]

def error(i):
    return hypothesis(houses[i], weights) - actual_values[i]

Squared error

We square it so that (1) all values are positive, preventing underestimates and overestimates from canceling each other out; and (2) larger errors are considered proportionally ‚Äúmore erroneous‚Äù than smaller errors.

$$ \begin{align} \textrm{SE}_i &= \textrm{error}^2\\ &= (\hat{Y}_i - Y_i)^2\\ &= (h_Œ∏(x^i) - Y_i)^2 \end{align} $$
def se(i):
    return error(i) ** 2

Mean squared error

The mean value of the squared error for all samples can give us an idea about our line‚Äôs goodness of fit.

$$ \begin{align} \textrm{MSE} &= \frac{\textrm{SE}_\color{red}{1} \color{blue}{+ \cdots + } \textrm{ SE}_\color{red}{\textrm{number of samples}}}{\textrm{number of samples}}\\ &= \frac{\color{blue}{\sum}_{\color{red}{i=1}}^{\color{red}{m}} \textrm{SE}}{m}\\ &= \frac{\sum_{\color{red}{i=1}}^{\color{red}{m}} (\hat{Y}_{\color{red}i} - Y_{\color{red}i})^2}{m}\\ &= \frac{1}{m} \sum_{\color{red}{i=1}}^{\color{red}{m}} (\hat{Y}_{\color{red}i} - Y_{\color{red}i})^2\\ &= \frac{1}{m} \sum_{\color{red}{i=1}}^{\color{red}{m}} (h_Œ∏(x^{\color{red}i}) - Y_{\color{red}i})^2 \end{align} $$
m = len(houses)
total_se = 0
for i in m:
    total_se += se(i)
mse = total_se / m

# More Pythonic and more similar to the actual math notation
mse = sum([se(house) for house in houses]) / len(houses)

Note: MSE gives us an indication of goodness of fit, but it‚Äôs difficult to tie that value directly to the data. You can use the RMSE (root mean squared error), which is just the square root of the MSE, to reframe the average error in terms of the data. In this case, the RMSE would tell us how much (in dollars) that our prediction line was off by.

Let‚Äôs use the mean squared error to rewrite this equation as a function of the collection of weights. Every time we change the weights, we will obtain a different line with a different goodness of fit (MSE), and this relationship can be illustrated by a function called the cost function.

def cost(weights):
    sum([se(house) for house in houses]) / len(houses)

Well, almost. This function is conventionally named \(J\):

$$ J(Œ∏) = \frac{1}{\color{magenta}{2}m} \sum_{i=1}^m (h_Œ∏(x^i) - Y_i)^2 $$

Where did that \(\color{magenta}{2}\) come from? Again, this is just a matter of convention. The \(\color{magenta}{2}\) will cancel out in the the next step.

def cost(weights):
    1 / (2 * len(houses)) * sum([se(house) for house in houses])

Keep in mind that \(Œ∏\) and \(Y\) are lists and \(x\) is a list of lists. What this means is that in a situation with two features (plus the dummy feature),

$$ \begin{align} h_{Œ∏_\color{teal}{0}, Œ∏_\color{teal}{1}, Œ∏_\color{teal}{2}}(x) &= h_{Œ∏_\color{teal}{0}}(x_\color{teal}{0}) + h_{Œ∏_\color{teal}{1}}(x_\color{teal}{1}) + h_{Œ∏_2}(x_\color{teal}{2}) \\ h_{Œ∏_\color{teal}{0}, Œ∏_\color{teal}{1}, Œ∏_\color{teal}{2}}(x^\color{red}{i}) &= h_{Œ∏_\color{teal}{0}}(x_\color{teal}{0}^\color{red}{i}) + h_{Œ∏_\color{teal}{1}}(x_\color{teal}{1}^\color{red}{i}) + h_{Œ∏_\color{teal}{2}}(x_\color{teal}{2}^\color{red}{i}) \\ J(Œ∏_\color{teal}{0}, Œ∏_\color{teal}{1}, Œ∏_\color{teal}{2}) &= \frac{1}{2m} \sum_{\color{red}{i=1}}^\color{red}{m} \Big[h_{Œ∏_\color{teal}{0}}(x_\color{teal}{0}^\color{red}{i}) + h_{Œ∏_\color{teal}{1}}(x_\color{teal}{1}^\color{red}{i}) + h_{Œ∏_\color{teal}{2}}(x_\color{teal}{2}^\color{red}{i}) - Y_{\color{red}i}\Big]^2 \end{align} $$

Numbers in teal represent feature numbers; numbers in red represent sample numbers.

Multivariable calculus: How much effect does each weight have?

Now, imagine each feature as knobs on a radio. Increasing or decreasing the weight of each feature is like turning up or down the knob for that feature. We want to ‚Äútune‚Äù our line to be as close to the data as possible by ‚Äúdialing‚Äù the features up and down. In order to do this, we need to determine the effect that a given combination of knob settings has on the final output.

In math terms, this is akin to asking ‚ÄúHow much does \(J\) change when \(Œ∏\) changes?‚Äù Sounds like derivatives from high school calculus.

$$ \begin{align} f(x) &= 5x^2 \\ \frac{df}{dx} &= 5\cdot2x^{2-1} \\ &= 10x \end{align} $$

Our function \(J\) is actually one function inside of another, so the chain rule applies. Bonus points if you remember that from high school‚ÄîI didn‚Äôt.

$$ \begin{align} J(Œ∏) &= \color{purple}{\frac{1}{2m}} \color{purple}{\sum_{i=1}^m} \color{purple}{\Big[} \color{orange}{h_Œ∏(x^i)} \color{purple}{- Y_i)\Big]^2} \\ \frac{dJ(Œ∏)}{dŒ∏} &= d_\color{purple}{\textrm{outer}} \cdot d_\color{orange}{\textrm{inner}} \end{align} $$
$$ \require{cancel} \begin{align*} \color{purple}{\textrm{outer}} &= \color{purple}{\frac{1}{2m}} \color{purple}{\sum_{i=1}^m} \color{purple}{(} \color{orange}{\textrm{inner}} \color{purple}{- Y_i)^2} & \color{orange}{\textrm{inner}} &= \color{magenta}{Œ∏_0}\color{teal}{x_0} + \color{magenta}{Œ∏_1}\color{teal}{x_1} + \cdots + \color{magenta}{Œ∏_n}\color{teal}{x_n} \\ d_\color{purple}{\textrm{outer}} &= \color{purple}{\frac{\cancel{2}}{\cancel{2}m}} \color{purple}{\sum_{i=1}^m} \color{purple}{\Big[} \color{orange}{h_Œ∏(x^i)} \color{purple}{{- Y_i\Big]}^2} & d_\color{orange}{\textrm{inner}} &= ??? \end{align*} $$

This is where I got stuck. \(Œ∏\) is a collection of values, not just a single value. Each knob on our radio affects the output individually, and we have to determine the individual effect of each knob.

It helps to start by breaking down what the chain rule is actually saying.

$$ \frac{d[\color{purple}{\textrm{outer}}(\color{orange}{\textrm{inner}}(x))]}{dx} = \frac{d_\color{purple}{\textrm{outer}}}{d_\color{orange}{\textrm{inner}}} \cdot \frac{d_\color{orange}{inner}}{dx} $$

This means our ‚Äúouter derivative‚Äù \(d_\color{purple}{\textrm{outer}}\) tells us how much our cost function \(J(Œ∏)\) changes in response to a given change in our hypothesis \(h(x)\). We now need to find the ‚Äúinner derivative‚Äù \(d_{\color{orange}{\textrm{inner}}}\), which tells us how much our hypothesis \(h(x)\) changes in response to a given change in our weights \(Œ∏\).

But since \(Œ∏\) is a collection of values, there isn‚Äôt a single derivative, but rather several partial derivatives, which indicate how much our hypothesis \(h(x^i)\) for a specific sample (house) \(x^i\) changes in response to a given change in each of the weights \(Œ∏_j\).

Note: Another labeling convention‚Äîjust as \(i\) is used to refer to, or ‚Äúindex,‚Äù samples from \(1\) through \(m\), the total number of samples, lowercase \(j\) is used to index features from \(0\) through \(n\), the total number of features. To return to our Einstein notation,

$$ x^{1 \> \leq \> i^\textrm{th} \textrm{ sample} \> \leq \> m \textrm{ samples}}_{0 \> \leq \> j^\textrm{th} \textrm{ feature} \> \leq \> n \textrm{ features}} $$

In math notation, this is written with a funny ‚Äúd‚Äù called a ‚Äúdel,‚Äù \(\partial\), like this:

$$ \frac{\partial h_Œ∏(x^i)}{\partial Œ∏_j} $$

This looks crazy, but the process of finding these partial derivatives is really the same as finding a normal derivative, except you treat all the other variables as constant, effectively ignoring them. So, for we now only have to concern ourselves with

$$ h_{Œ∏_j}(x^i) = \begin{cases} \color{magenta}{Œ∏_0}\color{teal}{x_0^i} \color{lightgray}{+Œ∏_1x_1^i + Œ∏_2x_2^i} & \text{when } j=0 \\ \color{lightgray}{Œ∏_0x_0^i} \color{lightgray}{+} \color{magenta}{Œ∏_1}\color{teal}{x_1^i} \color{lightgray}{+Œ∏_2x_2^i} & \text{when } j=1 \\ \color{lightgray}{Œ∏_0x_0^i + Œ∏_1x_1^i} \color{lightgray}{+} \color{magenta}{Œ∏_2}\color{teal}{x_2^i} & \text{when } j=2 \\ \end{cases} $$

The derivative of a variable times something else is just the something else. (For a line \(y = 2x\), \(\frac{dy}{dx}\) is just \(2\), since its slope will be 2 at every point along the line.) Thus,

$$ \frac{\partial h_Œ∏(x)}{\partial Œ∏_j} = \begin{cases} \frac{\partial h_Œ∏(x)}{\partial Œ∏_0} &= \color{lightgray}{Œ∏_0} \color{teal}{x_0^i} & \text{when } j=0 \\ \frac{\partial h_Œ∏(x)}{\partial Œ∏_1} &= \color{lightgray}{Œ∏_1} \color{teal}{x_1^i} & \text{when } j=1 \\ \frac{\partial h_Œ∏(x)}{\partial Œ∏_2} &= \color{lightgray}{Œ∏_2} \color{teal}{x_2^i} & \text{when } j=2 \end{cases} $$

Or just

$$ \frac{\partial h_Œ∏(x)}{\partial Œ∏_j} = \left\{\begin{array}{lr} x_0^i & \text{when } j=0 \\ x_1^i & \text{when } j=1 \\ x_2^i & \text{when } j=2 \end{array}\right\} = \color{red}{x_j^i} $$

That‚Äôs \(d_{\color{orange}{\textrm{inner}}}\).

$$ \begin{align} \frac{\partial{J(Œ∏)}}{\partial{Œ∏_j}} &= d_\color{purple}{\textrm{outer}} \cdot d_\color{orange}{\textrm{inner}} \\ &= \frac{1}{m} \sum_{i=1}^m (h_Œ∏(x^i) - Y_i) \cdot \color{red}{x_j^i} \end{align} $$

Phew! That was a lot of abstract math. Finally, we have something that can be translated into code.

def effect_of_weight(which_weight):
    return (1 / m) * sum([error(house) * weights[which_weight] for house in houses])

effect_of_all_weights = [effect_of_weight(j) for j in weights]

Stochastic gradient descent: Making regression lines fit again

Strictly speaking, \(\frac{1}{m} \sum_{i=1}^m (h_Œ∏(x^i) - Y_i) \cdot x_j^i\) is not a derivative, but a gradient‚Äîa collection of partial derivatives. In high school calculus, the derivative at a given point is visualized as the line that is tangent to the graph‚Äôs curve at that point. In multivariable calculus, the gradient at a given point is visualized as the plane that is tangent to the graph‚Äôs surface at the point.

In more concrete terms, imagine running a small piece of cardboard around the sides of a coffee mug so that the cardboard follows the curvature of the mug. Every point on the surface of the mug corresponds to some combination of weights, and the closer we are to the top of the mug, the greater the value of our cost function is, and so the more inaccurate our prediction is. We want to find the bottom of the mug, where the piece of cardboard is parallel to the ground, because that is where the value of the cost function is as low as possible.

When that value is zero, the line would fit our data perfectly. However, that‚Äôs not possible for real-world data, so we will settle for the lowest value‚Äîthat is, we want to minimize the cost function.

$$ \underset{Œ∏}{\arg\min} \, J(Œ∏) $$

In the language of math (and neural networks), this is called stochastic gradient descent.

  • The gradient is the thing we‚Äôre trying to minimize.
  • This process is stochastic (random) because we start with random weights (all zeros), which puts us at a random point on the mug.
  • It is a descent because we want to move down to a progressively flatter region of the mug with each attempt (combination of weights).

The descent occurs in ‚Äústeps.‚Äù Imagine, for a moment, a basic parabola \(f(x) = x^2\) instead of a mug. The derivative at any point \(x\) is \(2x\). Positive \(x\) values give us positive derivatives and negative \(x\) values give us negative derivatives. If we started at some point to the right of 0 and wanted to follow the parabola to its trough, we could do that by subtracting something from \(x\). Likewise, if we started at some point to the left of 0, we‚Äôd want to add something to \(x\)‚Äîor rather, subtract a negative value.

This means that if we start at any point \(x\) and subtract \(\frac{dy}{dx}\), we will tend toward the trough. We don‚Äôt necessarily know exactly what our new \(x\) value will be, but we can assume that subtracting \(\frac{dy}{dx}\) again will take us closer to the trough, although slightly less closer. Each step brings us increasingly closer but in progressively smaller steps. At some point, we will reach convergence, or a point that is close enough to minimum.

The same applies to gradients. The gradient for any set of weights \(Œ∏\) tells us the opposite direction we should go in to find the bottom of the mug. That means that if we start with some initial collection of weights \(Œ∏\) and keep subtracting the gradient, which is notated \(\nabla J\), we should eventually arrive at the bottom.

$$ \textrm{repeat } Œ∏ := Œ∏ - \nabla J \textrm{ until convergence} $$

Here is the generalized form of linear regression before and after vectorization, followed by the vectorized NumPy version.

This implies the equation is true for each \(i\), but the vectorized version automatically includes every sample at the same time without us even having to know how many there are. So let‚Äôs use the plural hypotheses to name the result.
  2. Look at how simple the math expression and the code become! (Of course, it does rely on a sufficient understanding of matrix multiplication.)
$$ \begin{align} h_Œ∏(x^i) &= \color{magenta}{Œ∏_0}\color{teal}{x_0^i} + \color{magenta}{Œ∏_1}\color{teal}{x_1^i} + \cdots + \color{magenta}{Œ∏_n}\color{teal}{x_n^i} \\ h_Œ∏(\textbf X) &= \textbf X\vec Œ∏ \end{align} $$

Vectorizing the cost function: So fresh, so clean

This makes it easy to express the error as the difference between the hypothesis and the actual value:

$$ \begin{align} \textrm{error} &= \hat{Y}_i - Y_i \\ &= h_Œ∏(x^i) - y_i \\ \vec e &= \textbf X\vec Œ∏ - \vec y \end{align} $$
errors = hypotheses - y

Recall that the cost function involves the sum of squared errors. In linear algebra, summation can be expressed as the product of a transposed vector of ones and a vector with the values to be summed, which struck me as a very clever manipulation.

$$ \begin{align} \vec o = \begin{bmatrix} 1 \\ 1 \\ \vdots \\ 1 \end{bmatrix} & \quad{} \vec e = \begin{bmatrix} e_1 \\ e_2 \\ \vdots \\ e_m \end{bmatrix} = \begin{bmatrix} h_Œ∏(x^1) - y_1 \\ h_Œ∏(x^2) - y_2 \\ \vdots \\ h_Œ∏(x^m) - y_m \end{bmatrix} \end{align} $$
$$ \begin{align} \vec o^T\vec e &= \begin{bmatrix} 1 & 1 & \cdots & 1 \end{bmatrix} \begin{bmatrix} e_1 \\ e_2 \\ \vdots \\ e_m \end{bmatrix} \\ &= \color{lightgray}1 \cdot e_1 + \color{lightgray}1 \cdot e_2 + \cdots + \color{lightgray}1 \cdot e_m = \sum_{i=1}^m e_i \end{align} $$

The cost function then becomes:

$$ \begin{align} J(Œ∏) &= \frac{1}{2m}\sum_{i=1}^m \Big[h_Œ∏(x^i) - Y_i\Big]^2 \\ &= \frac{1}{2m}\sum_{i=1}^m {(e_i)}^2 \\ &= \frac{1}{2m} \vec o^T \vec e^2 \end{align} $$
num_samples = X.shape[0]
o = np.ones((X.shape[0], 1))
cost = (o.T @ np.square(errors)) / (2 * num_samples)

At this point, it‚Äôs helpful to turn our cost function into a Python function that takes \(\textbf X, \vec y, \vec Œ∏\) as its inputs. This will make it easier to evaluate our model later.

def cost(X, y, weights):
    num_samples = X.shape[0]
    o = np.ones((num_samples, 1))
    errors = X @ weights - y
    cost_array = (o.T @ (np.square(errors))) / (2 * num_samples) # 1x1 matrix
    return cost_array[0][0] # plain number

Vectorizing the gradient: Ninja mode on overdrive

On to the gradient. This is where linear algebra really kicks this thing into high gear.

(This is also where I get to show off my LaTeX chops.)

$$ \begin{align} \nabla J = \frac{\partial J(Œ∏)}{\partial Œ∏_j} &= \Bigg\{\frac{\partial J(Œ∏)}{\partial Œ∏_0}, \frac{\partial J(Œ∏)}{\partial Œ∏_1}, \cdots, \frac{\partial J(Œ∏)}{\partial Œ∏_n}\Bigg\} \\ &= \frac{1}{m} \sum_{i=1}^m \Big[h_Œ∏(x^i) - Y_i\Big]^2 \cdot x_j^i \qquad{} \textrm{for } 0 \leq j \leq n \\ &= \frac{1}{m} \sum_{i=1}^m {(e_i)}^2 x_j^i \qquad{}\qquad{}\qquad{}\qquad{} „Äû„Äû\\ &= \frac{1}{m} \sum_{i=1}^m x_j^i {(e_i)}^2 \qquad{}\qquad{}\qquad{}\qquad{} „Äû„Äû\\ &= \Bigg\{ \color{teal}{ \frac{1}{m} \sum_{i=1}^m x_0^i {(e_i)}^2, \frac{1}{m} \sum_{i=1}^m x_1^i {(e_i)}^2, \cdots, \frac{1}{m} \sum_{i=1}^m x_n^i {(e_i)}^2 }\Bigg\} \end{align} $$

Transposing \(\textbf X\) and squaring \(\vec e\) gives us:

$$ \begin{align} \textbf X^T = \begin{bmatrix} x_0^1 & x_0^2 & \dots & x_0^m \\ x_1^1 & x_1^2 & \dots & x_1^m \\ \vdots & \vdots & \ddots & \vdots \\ x_n^1 & x_n^2 & \dots & x_n^m \end{bmatrix} & \quad{} \vec e^2 = \begin{bmatrix} {(e_1)}^2 \\ {(e_2)}^2 \\ \vdots \\ {(e_m)}^2 \end{bmatrix} \end{align} $$
$$ \textbf X^T \vec e^2 = \begin{bmatrix} x_0^1 {(e_1)}^2 + x_0^2 {(e_2)}^2 + x_0^3 {(e_3)}^2 + \cdots + x_0^m {(e_m)}^2 \\ x_1^1 {(e_1)}^2 + x_1^2 {(e_2)}^2 + x_1^3 {(e_3)}^2 + \cdots + x_1^m {(e_m)}^2 \\ \vdots \\ x_n^1 {(e_1)}^2 + x_n^2 {(e_2)}^2 + x_n^3 {(e_3)}^2 + \cdots + x_n^m {(e_m)}^2 \end{bmatrix} = \begin{bmatrix} \sum_{i=1}^m x_0^i{(e_1)}^2 \\ \sum_{i=1}^m x_1^i{(e_1)}^2 \\ \vdots \\ \sum_{i=1}^m x_n^i{(e_1)}^2 \\ \end{bmatrix} $$

Notice how multiplying this result by \(\frac{1}{m}\) gives us a vector containing the same values highlighted above in teal.

$$ \frac{1}{m} \textbf X^T \vec e^2 = \frac{1}{m} \begin{bmatrix} \sum_{i=1}^m x_0^i{(e_1)}^2 \\ \sum_{i=1}^m x_1^i{(e_1)}^2 \\ \vdots \\ \sum_{i=1}^m x_n^i{(e_1)}^2 \\ \end{bmatrix} = \begin{bmatrix} \color{teal}{ \frac{1}{m} \sum_{i=1}^m x_0^i{(e_1)}^2 } \\ \color{teal}{ \frac{1}{m} \sum_{i=1}^m x_1^i{(e_1)}^2 } \\ \color{teal}{ \vdots } \\ \color{teal}{ \frac{1}{m} \sum_{i=1}^m x_n^i{(e_1)}^2 } \\ \end{bmatrix} = \begin{bmatrix} \frac{\partial J(Œ∏)}{\partial Œ∏_0} \\ \frac{\partial J(Œ∏)}{\partial Œ∏_1} \\ \vdots \\ \frac{\partial J(Œ∏)}{\partial Œ∏_n} \end{bmatrix} $$

Astonishingly, that gigantic mess can be expressed as

$$ \nabla J = \frac{1}{m} \textbf X^T \vec e^2 $$
num_samples = X.shape[0]
gradient = (X.T @ np.square(errors)) / num_samples

Finally, we can work out the last function, the gradient descent function:

$$ \vec Œ∏ := \vec Œ∏ - Œ±\frac{1}{m} \textbf X^T \vec e^2 \\ \textrm{repeat until convergence} $$

Hey, where‚Äôd that \(Œ±\) come from? That‚Äôs the learning rate, a small number that adjusts the size of each training step. Too large and you jump right over the minimum; too small and you never reach the minimum.

For now, let‚Äôs choose an arbitrary value for \(Œ±\) and disregard the whole bit about convergence. If we wanted to perform stochastic gradient descent with 100 steps, this is how we‚Äôd do it:

for i in range(100):
    learning_rate = 0.01
    errors = X @ weights - y
    num_samples = X.shape[0]
    step_distance = (X.T @ errors) / num_samples
    weights -= learning_rate * step_distance

Let‚Äôs refactor this as a function gradient_descent that takes \(\textbf X, \vec y\), and the number of steps (training epochs) as its inputs, and outputs the weights \(\vec Œ∏\). Along the way, let‚Äôs have it tell us the cost. If all goes well, we should see that number approach zero as training progresses.

def gradient_descent(X, y, epochs, learning_rate=0.01):
    weights = np.zeros((X.shape[1], 1))
    num_samples = X.shape[0]
    for i in range(epochs):
        errors = X @ weights - y
        step_distance = (X.T @ errors) / num_samples
        weights -= learning_rate * step_distance
        print(cost(X, y, weights)[0][0]) # dot products are single values, but NumPy returns them as 1x1 matrices
    return weights

weights_300_epochs = gradient_descent(X, y, 300)

import sklearn.datasets

X, y = sklearn.datasets.make_regression(n_samples=500, n_features=5)

X = np.hstack((np.ones((X.shape[0], 1)), X)) # padding for bias column
y = np.vstack(y) # this just fixes a quirk of sklearn's output

weights = np.zeros((X.shape[1], 1))

def cost(X, y, weights):
    num_samples = X.shape[0]
    o = np.ones((num_samples, 1))
    errors = X @ weights - y
    cost_array = (o.T @ np.square(errors)) / (2 * num_samples)
    return cost_array[0][0]

def gradient_descent(X, y, epochs, learning_rate=0.01):

        step_distance = (X.T @ errors) / num_samples
        weights -= learning_rate * step_distance
    print(f"Completed {epochs} epochs of training.")
    print(f"Final cost: {cost(X, y, weights)}")
